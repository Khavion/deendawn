How it's built

Principles that shape the code

The “no-AI zone”

Quran text, translations and classical sources are never written or altered by the model. They enter only through a content pipeline, byte-for-byte from pinned sources, each with a SHA-256 checksum that fails the build on mismatch. Guard hooks physically block edits to religious text.

Two-tier “Ask”, hard-constrained

Tier A is deterministic search only — exact counts and verse lists, phrased as verifiable facts, ruling questions redirected to scholars. Tier B is an optional local-only AI that may paraphrase only the retrieved passages, with citations, and ships off behind a flag until scholar sign-off.

Privacy as an invariant

No accounts, no analytics, no ad SDKs. Location stays on the device. The only network calls are audio streaming and optional model downloads. An offline test suite proves every worship feature works with all servers stopped.

Tested against real data

1,680 prayer-time fixtures across cities, seasons and methods; the app's real SQL runs against the actual shipped databases; copy audits fail the build if the tip jar ever drifts into charity framing. 255 tests in all.

Honest ledger

Known limitations & technical debt

Recitation audio

The streaming player is complete but plays a labeled placeholder tone until license-cleared recordings exist. Popular reciters are not free to use — recommendation is written permission.

On-device AI is inert

Native stack compiles and the UI is wired, but model files are unuploaded and inference latency/quality are unmeasured. Ships off behind a flag.

Placeholder translation

A public-domain 1930 translation, watermarked; the shipping translation is a human decision.

Simulator-only checks

Real magnetometer, push delivery, purchase receipts, cold-start and 60fps timing are on a physical-device checklist.

No widgets / watch / OTA

A prayer-countdown Home Screen widget, Live Activity, and update strategy are unbuilt — obvious candidates.

No telemetry by design

Privacy-first means zero analytics — so improving the product without tracking is an open problem.

The ask

Where research would help most

Concrete questions for a technical research pass — the answers would directly improve the app.

Quran reading

  • Tajweed color-coding & word-by-word data with clear licensing, rendered at 60fps
  • Mushaf-accurate fixed-page layout feasibility
  • Better OFL Uthmani fonts than Amiri for ligature fidelity

On-device AI

  • Best small instruct models for citation-grounded paraphrase on an iPhone (Qwen3 vs alternatives, Q4/Q5 trade-offs, tokens/sec)
  • Arabic-aware embedding + hybrid retrieval that maximizes citation precision
  • Reliable on-device refusal for ruling-seeking / out-of-corpus queries

Audio

  • License-cleared full-Quran recitation sets usable in a free app with a tip jar — reciter, host, exact permission URL
  • Word-highlighted playback synced to verse timings, and its data licensing

Platform

  • Prayer-countdown widget + Live Activity + Watch complication cost in Expo/RN without ejecting
  • An OTA update strategy that never bypasses the content-checksum pipeline

Product & growth

  • How privacy-first apps learn what to improve with zero telemetry
  • Guideline 4.3(b) differentiation + ASO for a no-ads Islamic app
  • Monetization headroom that stays inside Apple 3.2.1 for a non-charity

Engineering

  • Supply-chain hardening for R2 model/audio downloads beyond SHA-256
  • Cleaner alternatives to the dual-SQLite + from-source-RN + fmt patches; what breaks at SDK 55/56
  • On-device performance-regression testing (cold start, scroll fps) in CI
